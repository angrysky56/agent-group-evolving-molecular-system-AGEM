Create src/orchestrator/llm_map.ts implementing ORCH-02 (parallel task dispatch with context preservation) per 05-RESEARCH.md Pattern 3:

1. AsyncLocalStorage import:
   - Import { AsyncLocalStorage } from 'node:async_hooks'
   - Export const contextStorage = new AsyncLocalStorage<Map<string, unknown>>()
   - This allows context to propagate through async boundaries

2. llm_map function signature:
   ```typescript
   export async function llm_map<T>(
     tasks: readonly Task<T>[],
     workerScriptPath: string,
     poolSize: number = 4
   ): Promise<readonly TaskResult<T>[]>
   ```

3. Implementation (per research Pattern 3):
   a. Create worker pool:
      - Create Array.from({ length: poolSize }, () => new Worker(workerScriptPath))
      - Workers are created fresh for each llm_map call (clean state)

   b. Dispatch all tasks in parallel:
      - For each task at index i:
        - Assign to worker = workers[i % poolSize] (round-robin)
        - Create a Promise that resolves when worker returns taskId matching task.id
        - On worker 'message' event, check if message.taskId === task.id
        - If match, resolve with TaskResult, unsubscribe handler
        - On worker 'error' event, resolve with { taskId: task.id, success: false, error }

   c. Context preservation:
      - Before postMessage to worker: get context = contextStorage.getStore()
      - Convert context to plain object: context ? Object.fromEntries(context) : {}
      - postMessage({ task, context: contextData })
      - Workers receive and can use context for logging/tracing

   d. Await all in parallel:
      - const results = await Promise.all(taskPromises)
      - Results may be out of original order (depends on worker completion order)

   e. Sort results by original task order:
      - Sort results by finding original task index:
        ```typescript
        return results.sort((a, b) => {
          const indexA = tasks.findIndex(t => t.id === a.taskId);
          const indexB = tasks.findIndex(t => t.id === b.taskId);
          return indexA - indexB;
        });
        ```

   f. Cleanup (finally block):
      - Terminate all workers: workers.forEach(w => w.terminate())
      - No exceptions should escape — worker termination is best-effort

4. Helper function: formatTaskForWorker(task: Task):
   - Deep clone task.payload ensuring all nested objects are plain (no class instances)
   - Validate no functions, circular references, or complex types
   - Document: "Workers can't deserialize functions or methods; serialize data only"

5. Error handling:
   - Worker errors (crash, unhandled exception): resolve with { taskId, success: false, error }
   - Task dispatch errors (bad task ID): resolve with error, don't reject llm_map()
   - All errors are collected and returned; llm_map() never rejects (unless workers crash before any task resolves)

6. JSDoc examples in comments:
   ```typescript
   // Example: Dispatch 3 parallel tasks
   // const results = await llm_map([
   //   { id: 'task-1', payload: { prompt: 'A' } },
   //   { id: 'task-2', payload: { prompt: 'B' } },
   //   { id: 'task-3', payload: { prompt: 'C' } },
   // ], new URL('./workers/TaskWorker.ts', import.meta.url).pathname, 2);
   // Results are in [task-1, task-2, task-3] order regardless of worker completion order
   ```

Key implementation notes (per 05-RESEARCH.md Pattern 3):
- Order preservation is CRITICAL: sort by original task index before returning
- Context serialization: plain objects only, no functions
- Per-task Promise handler: store handler reference so we can unsubscribe if needed
- Worker pooling: reuse workers[i % poolSize] across tasks (stateless from llm_map perspective)
- No retry logic: caller decides retry strategy
- Partial failures: return both successes and errors in results array

Create src/orchestrator/workers/TaskWorker.ts — the ESM worker script that receives task messages from the main thread and sends back results:

1. File structure (ESM worker):

   ```typescript
   import { contextStorage } from '../llm_map.js';

   ```

2. Message handler on parentPort:
   - parentPort.on('message', async (message: { task: Task<unknown>; context: Record<string, unknown> }) => { ... })
   - For each received message:
     a. Extract task and context from message
     b. Run the task in the worker's async context:
        - Use contextStorage.run(new Map(Object.entries(context)), async () => { ... })
        - This makes context available via contextStorage.getStore() for the duration of the task

     c. Try to execute task:
        - Construct a result object: { taskId: task.id, success: true, result: /* task output */ }
        - For now (STUB): no actual LLM call; just echo back a mock result
        - Result example: if task.payload has { prompt: string }, return a simple string result
        - Or: if task.payload is { value: number }, return value * 2 (deterministic stub)

     d. Send result back to main thread: parentPort.postMessage(result)

     e. Handle errors:
        - If task execution throws, catch and send: { taskId: task.id, success: false, error: new Error(...) }
        - Error must be serializable (string message, not full stack)

3. Error handling:
   - Worker should NOT crash on individual task failures
   - Errors are caught, wrapped, and sent as TaskResult back to main thread
   - parentPort.on('error') is the worker uncaught error handler (if everything fails)

4. Graceful shutdown:
   - Worker process exits naturally when parentPort closes (main thread terminates)
   - No explicit shutdown needed

5. Typing:
   - Use any or unknown for task.payload since it's generic
   - TaskResult<unknown> is acceptable (caller types the result when calling llm_map)

   - "Task execution is sandboxed; errors are caught and returned, not thrown"
   - "Context is available via contextStorage.getStore() during task execution"
